# Internationalization

KVT webapp uses i18next and react-i18next.

The setup follows the official i18next TypeScript guide:

- [Getting started](https://www.i18next.com/overview/getting-started)
- [API](https://www.i18next.com/overview/api)
- [TypeScript](https://www.i18next.com/overview/typescript)
- [Plugins and utils](https://www.i18next.com/overview/plugins-and-utils)

## Location

Translations live in the app core layer:

```text
src/core/i18n/
  config.ts
  i18next.d.ts
  resources.ts
  locales/
    ru/common.ts
    ru/chat.ts
    en/common.ts
    en/chat.ts
```

`app` bootstraps i18n, but translation ownership is not in `src/app`.

## Usage

Use the normal react-i18next API:

```ts
const { t } = useTranslation('chat')

t('title')
```

Autocomplete works because `i18next.d.ts` registers the resource shape.

## Adding a namespace

1. Add `ru/newNamespace.ts`.
2. Add `en/newNamespace.ts`.
3. Add the namespace to `resources.ts`.
4. Use `useTranslation('newNamespace')`.

## Tooling

VS Code is configured for [i18n Ally](https://github.com/lokalise/i18n-ally).

Recommended extension:

```text
lokalise.i18n-ally
```

The extension is configured to read:

```text
app/webapp/src/core/i18n/locales/{locale}/{namespace}.ts
```

The i18next docs also list `i18next-cli` for extraction, sync, lint, and type generation. We have
not added it yet because the current app keeps translations as typed TypeScript modules.
